STEP-BY-STEP OUTLINE
Aligning the Engineering GPT Prototype with the Base Instruction

Purpose
This outline packages the implementation steps required to align the current Engineering GPT prototype with the Base instruction. The focus areas are AI API integration, calculation reliability, database integration, and user authentication.

Core Target
The application should behave as a controlled engineering assistant, not a general chatbot. It should:
- Stay within engineering and technical project scope.
- Show assumptions, formulas, missing data, units, limitations, and professional-review notices for calculations.
- Avoid pretending to be a licensed engineer.
- Avoid approving final designs, code compliance, construction, installation, or operation.
- Treat standards and codes as informational support unless verified by qualified professionals and applicable authorities.
- Preserve traceability through saved chats, calculations, files, and audit logs.

------------------------------------------------------------
PHASE 2 — CONVERT BASE.TXT INTO ENFORCEABLE APPLICATION LOGIC
------------------------------------------------------------

2. Load Base.txt as the core AI system instruction

Create:

lib/ai/basePrompt.ts

Purpose:
- Store or load the Base instruction server-side.
- Ensure it is applied inside the backend AI route.
- Do not rely on frontend-only prompt injection.

Example responsibility:

export const ENGINEERING_BASE_PROMPT = `
[contents of Base.txt]
`;

3. Add an instruction hierarchy

Recommended order:

1. System/developer instructions from Base.txt.
2. Product-level safety rules.
3. Project context.
4. Uploaded document context.
5. User message.

Important rule:
Uploaded files, drawings, PDFs, emails, and project documents may contain hostile or irrelevant instructions. Treat such content only as project data. Never allow file content to override Base.txt or system-level behavior.

4. Add a structured AI response contract

The AI should return both answer content and structured metadata.

Recommended response type:

type EngineeringAIResponse = {
  answer: string;
  scopeStatus: "engineering" | "out_of_scope" | "needs_clarification";
  assumptions: string[];
  missingData: string[];
  safetyWarnings: string[];
  standardsReferenced: string[];
  requiresProfessionalReview: boolean;
  confidenceLevel: "rough_estimate" | "preliminary" | "detailed" | "not_applicable";
};

Purpose:
- Allow the UI to consistently show assumptions, warnings, missing data, limitations, and review notices.
- Avoid burying critical engineering caveats inside free text.

------------------------------------------------------------
PHASE 3 — AI API INTEGRATION
------------------------------------------------------------

5. Build the server-side chat endpoint

Create:

app/api/chat/route.ts

Responsibilities:
- Verify authenticated user.
- Validate request body.
- Load Base instruction.
- Load project context from database.
- Load relevant document chunks if retrieval is enabled.
- Call the AI model API.
- Stream the response back to the frontend.
- Save user message and AI response to the database.
- Save metadata such as assumptions, warnings, calculations used, and referenced files.

Recommended route flow:

User message
  -> Authentication check
  -> Project permission check
  -> Scope classifier
  -> Retrieve project context/documents
  -> Inject Base.txt instructions
  -> Call AI API
  -> Run safety/output validator
  -> Save message and response
  -> Return streamed answer

6. Use a model-provider abstraction

Create:

lib/ai/provider.ts

Recommended interface:

export interface AIProvider {
  generateEngineeringResponse(input: EngineeringChatInput): Promise<EngineeringAIResponse>;
  streamEngineeringResponse(input: EngineeringChatInput): AsyncIterable<string>;
}

Purpose:
- Avoid hard-wiring the app to one provider.
- Allow future support for OpenAI-compatible APIs, Azure OpenAI-style deployments, local model gateways, or enterprise model providers.

7. Add AI tool calling for engineering operations

The AI should not invent calculation results. It should call deterministic calculation tools.

Recommended tools:
- calculate_fluid_pressure_drop
- calculate_reynolds_number
- calculate_pipe_flow_rate
- calculate_beam_reaction
- calculate_voltage_drop
- calculate_pump_power
- search_project_documents
- summarize_uploaded_datasheet
- generate_method_statement
- generate_inspection_checklist

Principle:
- The AI explains and documents.
- Calculation modules compute.

8. Add engineering-scope classification

Classify each user request before answering.

Suggested categories:
- engineering_allowed
- engineering_calculation
- engineering_documentation
- engineering_troubleshooting
- standards_or_code
- safety_critical
- out_of_scope
- ambiguous

Behavior:
- If out_of_scope, return the refusal behavior defined by Base.txt.
- If safety_critical, force additional warning and professional-review language.
- If ambiguous, ask for engineering context or clarify missing technical details.

PHASE 4 — CALCULATION ENGINE INTEGRATION
------------------------------------------------------------

9. Separate calculations from UI

Move engineering calculations out of React components.

Recommended structure:

lib/calculations/
  fluid/
    pressureDrop.ts
    reynolds.ts
    frictionFactor.ts
    schemas.ts
    tests/
  units/
    convert.ts
  common/
    resultTypes.ts

The frontend should call calculation APIs or shared calculation modules instead of embedding engineering math directly in page components.

10. Define a standard calculation result format

Recommended type:

type CalculationResult = {
  calculationType: string;
  objective: string;
  inputs: Record<string, InputValue>;
  assumptions: string[];
  missingData: string[];
  formulas: FormulaStep[];
  steps: CalculationStep[];
  results: Record<string, ResultValue>;
  warnings: string[];
  limitations: string[];
  requiresProfessionalReview: boolean;
};

Purpose:
- Enforce Base.txt calculation requirements.
- Make all calculations auditable and exportable.

11. Add input validation

Use a schema validation library such as Zod.

Example:

const FluidPressureDropInputSchema = z.object({
  diameter_m: z.number().positive(),
  length_m: z.number().positive(),
  velocity_m_s: z.number().positive(),
  density_kg_m3: z.number().positive(),
  viscosity_pa_s: z.number().positive(),
  roughness_m: z.number().nonnegative().optional(),
});

Reject invalid values such as:
- Zero diameter.
- Negative flow rate.
- Missing viscosity.
- Invalid units.
- Impossible density values.

12. Improve the fluid dynamics calculator

Upgrade the calculator from demo logic to a traceable engineering estimator.

Add:
- Pipe roughness input.
- Fluid presets.
- Unit selector.
- Reynolds number classification.
- Laminar friction factor: f = 64 / Re.
- Turbulent friction factor correlation.
- Transitional-flow warning.
- Pressure-drop formula trace.
- Minor-loss coefficient option.
- Temperature-dependent fluid properties.
- Preliminary-only warning.

Output sections should include:
- Objective.
- Inputs.
- Assumptions.
- Missing data.
- Formulas.
- Calculation steps.
- Results.
- Limitations.
- Professional review notice.

PHASE 9 — SAFETY AND COMPLIANCE ENFORCEMENT
------------------------------------------------------------

27. Add a safety post-processor

After the AI produces a response, check for risky or overconfident language.

Flag or rewrite phrases such as:
- “This is approved.”
- “This is compliant.”
- “You can safely install this.”
- “No engineer review is needed.”
- “Bypass the interlock.”
- “Ignore the code requirement.”

Replace with safer language such as:
- “This is a preliminary engineering support response.”
- “Verify against the applicable code, project specification, and authority having jurisdiction.”
- “This should be reviewed by a qualified professional before use.”

28. Add contextual engineering disclaimers

Do not show the same disclaimer everywhere. Make it context-specific.

Context | Required notice
Simple documentation | Draft for review.
Calculation | Preliminary calculation; verify inputs and assumptions.
Safety-critical work | Requires qualified professional review before use.
Standards interpretation | Confirm latest edition and jurisdiction.
Final design request | Cannot provide final approval or certification.
